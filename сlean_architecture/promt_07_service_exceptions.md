# Этап 6 (дополнение): Кастомные исключения сервиса и их маппинг в Web API

### Цель

- Зафиксировать одноразовое создание кастомных исключений в слое `Services`
- Зафиксировать, где и как настраивается глобальный маппинг исключений в HTTP-ответы в Web API (middleware/exception handler)
- Правило проекта: исключения остаются в `Services` и используются в unit-тестах сервисного слоя; контроллеры доменные ошибки не ловят

---

## 1) Где живут исключения

- Слой: `src/ProjectName/Services/Exceptions`
- Исключения являются частью контрактов поведения сервисов (не HTTP-контроллеров)
- Сервис выбрасывает исключения; Web-слой (глобальный обработчик) преобразует их в HTTP статус-коды и JSON-ответ

---

## 2) Набор исключений и соответствие HTTP

### BadRequestException → 400

- Назначение: ошибки валидации входных данных и бизнес-валидации (когда запрос сформирован неверно или нарушает правила)
- Поддержка “мульти-ошибок”: если нужно вернуть несколько причин, исключение хранит `List<string> Messages`
- Рекомендация: для одного сообщения использовать `BadRequestException(string message)`, для набора ошибок — `BadRequestException(IEnumerable<string> messages)`

### UnauthorizedException → 401

- Назначение: отсутствие/невалидность критичных данных аутентификации/claims в тех местах, где сервисный слой это проверяет
- Примечание: 401/403 чаще всего формируются инфраструктурой авторизации в Web-слое, но проект допускает `UnauthorizedException` из сервисов как бизнес-гарантию и для тестов

### NotFoundException → 404

- Назначение: запрошенный ресурс не найден (например, `GetById` вернул `null`)
- Сообщение: из `EntityNameTexts.Messages.Error.*` (единый текстовый источник)

### ConflictException → 409

- Назначение: конфликт состояния (например, нельзя удалить сущность из-за зависимостей, уникальность, гонка состояний)

---

## 3) Одноразовое создание классов исключений (Services)

Создать в `src/ProjectName/Services/Exceptions`:

- `BadRequestException`
  - `public List<string> Messages { get; } = [];`
  - конструктор для одного сообщения
  - конструктор для набора сообщений (задаёт `base(...)` как общий текст “ошибки валидации”, а детали кладёт в `Messages`)
- `NotFoundException`
- `ConflictException`
- `UnauthorizedException`

---

## 4) Где настраивается маппинг исключений в Web API

### Требование

- Web-слой обязан иметь глобальный обработчик исключений, который:
  - маппит исключения `Services.Exceptions.*` в HTTP статус-коды (400/401/404/409)
  - возвращает единый JSON-формат ошибки для API
  - не требует try/catch в контроллерах для доменных ошибок

### Важный нюанс про UseExceptionHandler

- Если используется `IExceptionHandler` (типовой подход в современных версиях ASP.NET Core), то `UseExceptionHandler` должен быть включён в режиме, который действительно вызывает зарегистрированный обработчик:
  - правильно: `app.UseExceptionHandler();`
  - нежелательно для Web API: `app.UseExceptionHandler("/Error/500");` (path-based обработка может обходить вашу логику маппинга исключений)

---

## 5) Минимальная схема настройки (Web API)

### DI

- зарегистрировать глобальный обработчик исключений (например, `GlobalExceptionHandler`), который реализует `IExceptionHandler`
- при необходимости зарегистрировать отдельный маппер исключений (например, `ExceptionMapper`), который делает `Exception → (StatusCode, Message, Errors)`

### HTTP pipeline

- включить `UseExceptionHandler()` достаточно рано в pipeline, чтобы он перехватывал исключения контроллеров/сервисов

---

## 6) Контракт ответа об ошибке для API

- Проект выбирает один единый формат ответа об ошибке для всех исключений:
  - либо `ProblemDetails`
  - либо собственный DTO (например: `{ messageType, messageText, statusCode, errors }`)

В остальных promt-файлах считать, что:

- контроллеры не возвращают разношёрстные ошибки вручную
- формат ошибок и коды задаются глобально (п. 4–5)
