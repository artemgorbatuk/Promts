# Этап 1: Опрос перед реализацией (Requirements)

### Требования

- Перед созданием модели/конфигурации/контекста/миграции/репозитория/сервиса/контроллера ИИ обязан провести опрос и зафиксировать ответы
- Опрос должен покрывать: доменную модель, схему БД, поля и ограничения, связи и правила удаления, требования к list-операциям (поиск/фильтр/сортировка/пагинация), include-опции, контракты DTO, маршруты API, миграции
- Если ответов недостаточно для корректной реализации, ИИ обязан задать дополнительные вопросы до начала генерации кода

### Дефолтные ответы (если ответа нет — использовать их)

#### Общие

- Схема БД: `public`
- Первичный ключ: `int` identity, имя `Id`
- Таблица/DbSet/route: во множественном числе, по умолчанию `EntityName` → `EntityNames`
- Индексы: создавать только если нужна уникальность/поиск; по умолчанию без дополнительных индексов
- DeleteBehavior для связей: `Restrict` (если явно не нужна другая стратегия)
- Include по умолчанию: выключены; включать только если реально нужно для операции

#### List / QueryPipeline

- List-эндпоинт: `POST /list` и `[FromBody]` (для сложных фильтров/сортировок)
- Пагинация: `Skip = 0`, `Take = 50`, `MaxTake = 200`
- Дефолтная сортировка: `Id desc`
- String search/filter: ignore-case по умолчанию `true`
- Операторы: использовать набор из `QueryPipelineOperators` (включая `in/notin` только если они реализованы/разрешены)

#### DTO / сервис / репозиторий

- DTO list по умолчанию: минимально необходимые поля (`Id`, `Name`), без аудита
- Поля аудита (`CreatedOn/UpdatedOn/CreatedBy`): не добавлять, если явно не требуется
- Методы репозитория для list: `GetListAsync` + `CountAsync` (для total), `GetSingleOrDefaultAsync` для карточки
- Seed-данные: не добавлять, если явно не требуется
- Миграция: создавать только по явной просьбе (следуя `TaskXXX`)

### Список вопросов

#### 1) Общая информация по сущности

- Какое точное имя сущности (EntityName) в единственном числе на английском?
- Какое имя сущности во множественном числе (для таблицы/DbSet/route)?
- Какой это тип сущности: справочник, бизнес-объект, лог/аудит, join-таблица?

#### 2) Таблица и схема БД

- Какое точное имя таблицы (во множественном числе) и схема (если используется)?
- Какой тип первичного ключа: `int` (identity) / `Guid` / другое?
- Нужны ли альтернативные ключи (natural key) или composite key?

#### 3) Поля (колонки) и ограничения

Для каждого поля:
- Имя поля (EN) и назначение (что хранит)
- Тип: `string/int/long/decimal/bool/DateTime/Guid` (+ для строк: короткая строка или `text`)
- Nullable или Required?
- Для строк: `HasMaxLength` (конкретное число) или `text` без ограничения?
- Для чисел: точность/масштаб (например, `numeric(7,2)`), диапазоны/валидации
- Значение по умолчанию (если нужно) и где задаётся (DB default vs app)
- Нужны ли уникальность/индексы по этому полю?
- Можно ли менять поле после создания (для Update), или оно readonly?

#### 4) Индексы и уникальность

- Какие индексы нужны (уникальные/неуникальные)?
- По каким колонкам (одна/несколько), и как назвать индекс (`IX_...`)?
- Нужны ли “функциональные” индексы (например, lower(name)) для case-insensitive поиска?
- Нужны ли ограничения уникальности “в рамках родителя” (например, `ParentId + Name`)?

#### 5) Связи (FK) и навигации

Для каждой связи:
- Тип связи: 1:1, 1:N, N:N, self-referencing
- FK обязательный или nullable?
- Имя FK поля (`...Id`) и навигаций (`Entity`, `Entities`)
- Нужны ли коллекции навигаций, и должны ли они быть `virtual`?
- Поведение при удалении: `Restrict` / `SetNull` / `Cascade` и почему
- Нужно ли запрещать каскадное удаление в бизнес-сценарии?

Для N:N:
- Нужна ли явная join-таблица (имя, составной ключ, индексы)?
- Нужны ли дополнительные поля в join (например, `CreatedOn`, `SortOrder`)?

#### 6) Include-опции (что подгружаем в репозитории)

- Какие связи реально нужны при чтении:
  - для Info (карточка)
  - для List (таблица)
  - для DisplayCreate/DisplayUpdate (dropdown-данные)
- Нужны ли разные include-флаги в `*QueryOptions` (например, `IncludeRelatedEntities`, `IncludeOwner`, `IncludeTags`)?
- Есть ли риск N+1 и нужно ли использовать `Include/ThenInclude`?
- Для больших коллекций: нужна ли пагинация/фильтрация внутри include (обычно нет) или отдельные эндпоинты?

#### 7) List-операции (QueryPipeline: поиск/фильтр/сортировка/пагинация)

- Нужен ли list-эндпоинт для сущности (DisplayList/GetList)?
- Какие поля разрешены для:
  - Search (только строковые/проекции в строку)
  - Filter (строка/число/дата/bool/guid)
  - Order (какие поля сортируются)
- Нужны ли вычисляемые поля для списка (например, `CreatedDateOn` из `CreatedOn.Date`) и как их поддерживать в сортировке?
- Какой дефолтный порядок сортировки, если пользователь не выбрал (например, `Id desc`)?
- Параметры пагинации:
  - default `Take` и max `Take`
  - разрешён ли `Skip` без `Take`
- Нужны ли операторы `in/notin` (списки) и для каких типов (даты/числа/guid)?
- Нужна ли фильтрация boolean (eq/notequal) и формат значений из клиента (`true/false`)?
- Нужно ли ignore-case для строковых фильтров/поиска (по умолчанию true)?

#### 8) Контракты DTO (Services/Models)

- Какие поля возвращает:
  - DisplayList (колонки таблицы)
  - DisplayInfo (карточка)
  - DisplayCreate/DisplayUpdate (данные формы + dropdown items)
- Какие поля принимает Create/Update, какие обязательные?
- Как оформлять связанные сущности в DTO: через `...Ids` (selected ids) и отдельные `DropdownItems`?
- Нужны ли поля аудита (`CreatedOn`, `UpdatedOn`, `CreatedBy`) в DTO?

#### 9) Репозиторий и Unit Of Work

- Какие методы реально нужны сервису:
  - `GetSingleOrDefaultAsync`, `GetListAsync`, `CountAsync`
  - `Create/Update/Delete`, `GetNew`
- Нужны ли отдельные репозитории для связанных сущностей (dropdown-данные)?
- Нужны ли дополнительные `QueryOptions` кроме includes и QueryPipeline (например, фильтр по родителю)?

#### 10) Сервисный слой

- Какие операции нужны: Create/Update/Delete/Info/List/Dropdown?
- Нужны ли транзакции для Create/Update/Delete (обычно да)?
- Какие бизнес-валидации обязательны (помимо null/required)?
- Какие исключения использовать: NotFound/BadRequest и какие тексты сообщений?
- Что логировать на Start/Success/Error, какие сообщения в Texts?

#### 11) Контроллер / API

- Какие маршруты и HTTP-методы нужны?
- List-эндпоинт:
  - будет `POST /list` с body (рекомендуется для сложных фильтров) или `GET` с query?
  - нужна ли авторизация/политики на list/info/create/update/delete?
- Нужен ли отдельный endpoint для dropdown-данных?
- Какие коды ответов должны быть описаны в `ProducesResponseType`?

#### 12) Миграции

- Нужно ли создавать миграцию сейчас или это позже?
- Какой следующий номер TaskXXX (проверить существующие миграции)?
- Нужны ли seed-данные (начальные справочники) и где их хранить/накатывать?
